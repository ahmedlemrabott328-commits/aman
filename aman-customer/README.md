# AMAN Customer — تطبيق الزبون (Flutter)

تطبيق الزبون الكامل لمنصة AMAN، Material Design 3، بنفس الهوية البصرية وبنية Clean Architecture المعتمدة في [aman-captain](../aman-captain)، ويعمل مباشرة مع [aman-backend](../aman-backend).

## البنية

```
lib/
├── core/            # نفس بنية aman-captain: theme, network, storage, realtime, router, providers
└── features/
    ├── auth/         # OTP: هاتف → تحقق → دخول
    ├── request/       # الشاشة الرئيسية: خريطة + بحث عن وجهة + عرض سعر تقديري + طلب رحلة
    ├── trip/          # تتبع الرحلة لحظيًا + الإلغاء + التقييم بعد الانتهاء
    ├── history/       # سجل الرحلات
    └── profile/       # الملف الشخصي وتسجيل الخروج
```

## تدفق العمل الأساسي (يطابق القسم 4 من المواصفات)

1. **تسجيل الدخول (OTP):** نفس تدفق تطبيق الكابتن تمامًا.
2. **تحديد الانطلاق والوجهة:** الموقع الحالي يُحدَّد تلقائيًا كنقطة انطلاق عبر `Geolocator`؛ الوجهة تُختار عبر `DestinationSearchScreen` (بحث جغرافي فعلي عبر حزمة `geocoding`، وليس قائمة أماكن وهمية).
3. **عرض السعر التقديري:** فور اختيار الوجهة، تُحسب المسافة محليًا (`Geolocator.distanceBetween`) وتُرسَل مباشرة إلى `/trips/estimate` في aman-backend — نفس منطق التسعير الحقيقي المستخدَم عند إنشاء الرحلة فعليًا، وليس تقديرًا وهميًا في التطبيق.
4. **طلب الرحلة:** لا يختار الزبون كابتنًا (كما يفرض القسم 6)؛ الشاشة تنتقل لتتبع الرحلة فور الطلب.
5. **تتبع الكابتن لحظيًا:** `TripTrackingScreen` تشترك في قناة `trip.{id}` عبر Reverb؛ أيقونة الكابتن على الخريطة تتحرك فور كل `captain.location.updated`، وحالة الرحلة (قبول/وصول/بدء/إنهاء) تُحدَّث فوريًا دون Polling.
6. **الاتصال بالكابتن:** زر اتصال مباشر (`tel:`) بمجرد قبول كابتن للرحلة.
7. **الإلغاء:** متاح فقط أثناء `requested/searching/accepted` (يطابق `Trip::isCancellable()` في Backend).
8. **التقييم:** `RatingScreen` تظهر تلقائيًا فور اكتمال الرحلة (عبر `ref.listen` على حالة الرحلة).

## القيود المعروفة (بصراحة كاملة)

- **لا يوجد `GET /services` أو `GET /cities` عام في aman-backend.** الخدمات الثلاث (نقل/مطار/توصيل) والمدينة الافتراضية (نواكشوط) مُثبَّتة يدويًا في `request_controller.dart`، بنفس القيد الموثَّق سابقًا في لوحة الإدارة React. يجب إضافة هذين الـ endpoints قبل الإطلاق التجاري الفعلي، خصوصًا لدعم تعدّد المدن مستقبلًا.
- **البحث الجغرافي (`geocoding`) يعرض نتائج بإحداثيات فقط دون عنوان مقروء لكل نتيجة قبل الاختيار** (العنوان الكامل يُحسَب فقط بعد اختيار نتيجة عبر reverse geocoding)؛ تجربة مقبولة لكنها أقل سلاسة من بحث أماكن حقيقي (Google Places/Mapbox Search) — تحسين مستقبلي معقول.
- **مدة الرحلة التقديرية** (`durationMin`) تُحسب محليًا بافتراض سرعة متوسطة 28 كم/سا، وليس عبر محرك توجيه فعلي (OSRM/Google Directions) يحسب زمن الطريق الحقيقي مع الازدحام. يؤثر هذا فقط على دقة السعر التقديري المعروض قبل الطلب؛ السعر النهائي يُعاد احتسابه من القيم الفعلية عند إنهاء الرحلة على أي حال.
- **العناوين المحفوظة (`customer_addresses`)** موجودة في قاعدة البيانات لكن لا توجد بعد شاشة لإدارتها من التطبيق.
- لا توجد اختبارات بعد (Unit/Widget).

## التشغيل محليًا

```bash
flutter pub get

flutter run \
  --dart-define=API_BASE_URL=http://10.0.2.2:8000/api/v1/customer \
  --dart-define=BROADCAST_AUTH_URL=http://10.0.2.2:8000/broadcasting/auth \
  --dart-define=REVERB_HOST=10.0.2.2 \
  --dart-define=REVERB_PORT=8080 \
  --dart-define=REVERB_APP_KEY=<REVERB_APP_KEY من .env الخاص بـ aman-backend>
```
